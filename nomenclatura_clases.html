Nomenclatura para Clases, Métodos y Comentarios

Nomenclatura de Clases

Las clases son planos para crear objetos. Su nombre debe reflejar la entidad o el concepto que representan.

  • PascalCase: Es la convención más común para nombres de clases en la mayoría de los lenguajes orientados a objetos (Java, C#, Python, JavaScript). Cada palabra comienza con mayúscula (ej: `Usuario`, `GestorPedidos`, `HttpRequest`).
  • Los nombres de clases suelen ser sustantivos (ej: `Coche`, `Cliente`, `Servicio`).
  • Deben ser descriptivos y evitar abreviaciones a menos que sean muy comunes y reconocidas.

Nomenclatura de Métodos/Funciones

Los métodos (en clases) o funciones (fuera de clases) realizan acciones. Su nombre debe describir la acción que llevan a cabo.

  • camelCase: Común en Java, JavaScript, C# (para métodos privados/protegidos). Empieza en minúscula y las siguientes palabras en mayúscula (ej: `calcularTotal`, `guardarDatos`, `obtenerUsuarioPorId`).
  • snake_case: Común en Python (ej: `calcular_total`, `guardar_datos`).
  • PascalCase: A veces usado en C# para métodos públicos (ej: `CalcularTotal`).
  • Los nombres de métodos suelen comenzar con un verbo (ej: `get`, `set`, `calcular`, `guardar`, `validar`).

Uso y Nomenclatura de Comentarios

Los comentarios son explicaciones en el código. Deben usarse para aclarar "por qué" se hace algo, no simplemente "qué" se hace (ya que el código debería ser auto-explicativo si la nomenclatura es buena).

  • Comentarios de línea única: Suelen empezar con `//` (en C++, Java, JavaScript, C#) o `#` (en Python). Se usan para explicaciones cortas o para desactivar temporalmente una línea.
  • Comentarios de bloque: Encerrados entre `/* ... */` (en C++, Java, JavaScript, C#) o `''' ... '''` (en Python - docstrings). Se usan para explicaciones más largas, descripciones de funciones/clases (docstrings), o secciones de código.
  • Deben ser concisos, relevantes y mantenerse actualizados con el código. Un comentario desactualizado es peor que no tenerlo.
  • Evitar comentar cada línea. Comentar la lógica compleja o las decisiones de diseño importantes.
